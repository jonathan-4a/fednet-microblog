# API Docker deployment

This document describes how the **microblog API** (`api/`) is built into a Docker image and run—both as a single-container workflow and as a **multi-instance federation test network** driven by `api/docker/launch.sh`.

## Layout

| Path | Role |
|------|------|
| `api/docker/Dockerfile` | Multi-stage image: build bundle with Bun, run with production deps |
| `api/docker/docker-compose.yml` | **Generated or checked in** stack: `nginx` reverse proxy + N API instances |
| `api/docker/launch.sh` | Generates `docker-compose.yml`, `nginx.conf`, and `/etc/hosts` for N nodes; optional build/start |
| `api/docker/nginx.conf` | **Generated** by `launch.sh` (not hand-maintained when using the launcher) |
| `api/.dockerignore` | Excludes `node_modules`, `.env`, local DB files, etc. from the build context |

The Docker **build context** is the **`api/` directory** (`context: ..` from `api/docker/`), with `dockerfile: docker/Dockerfile`.

---

## Image build (`Dockerfile`)

The image uses **`oven/bun:1-alpine`** for both stages.

### Build stage (`AS build`)

1. **`WORKDIR /app`**
2. Copies **`package.json`** and **`bun.lock`**, then runs **`bun install --frozen-lockfile`** so installs are reproducible.
3. Copies the rest of the API source (**`COPY . .`**).
4. Runs **`bun run build`**, which (from `package.json`) executes:

   ```text
   bun build src/server.ts --outdir dist --target bun
   ```

   This emits a Bun-targeted bundle under **`/app/dist/`** (e.g. `dist/server.js`).
5. **`RUN ls -la /app/dist/`** is a sanity check that the bundle exists after build.

### Production stage (final image)

1. Again **`WORKDIR /app`** on a fresh `oven/bun:1-alpine` layer.
2. **`package.json` + `bun.lock`** and **`bun install --production --frozen-lockfile`** — only runtime dependencies (no devDependencies).
3. Copies artifacts from the build stage:
   - **`/app/dist`** → **`./dist`**
   - **`/app/public`** → **`./public`**
   - **`/app/openapi.json`** → **`./openapi.json`**
4. **`EXPOSE 3000`** documents the default port the image expects; **Compose overrides this** in practice (see below).
5. Default **`CMD`**: **`["bun", "run", "dist/server.js"]`** — same as the `start:prod` npm script.

The resulting image tag used by the federation launcher is **`microblog-built:latest`** (see `launch.sh`).

## What is excluded from the image (`.dockerignore`)

The build context excludes at least:

- `node_modules` (reinstalled inside the image)
- `.git`
- `.env` (secrets must be supplied at runtime, e.g. via Compose)
- `*.db`, `*.db-shm`, `*.db-wal` (local SQLite files)
- `logs`, `.DS_Store`, `coverage`

So the database file is **not** baked into the image; persistence is via **mounted volumes** when using Compose.

## Runtime: `server.ts` and environment

The bundled entrypoint runs **`dist/server.js`**, which:

- Sets **`UV_THREADPOOL_SIZE`** from CPU count.
- Reads **`PORT`** from the environment (**required**): `parseInt(process.env.PORT!, 10)`.
- Initializes the DB schema and admin user via **`composition-root`** (see env vars below).
- Starts **`Bun.serve`** with the Hono **`app.fetch`** handler.

Database configuration is centralized in **`src/connection.ts`**. For Docker federation setups, Compose sets **`DB_PATH`** (SQLite file under `/app/data/...`). See comments in that file for **`DB_DIALECT`**, **`DATABASE_URL`**, and **`DB_PATH`**.

Relevant **application** environment variables used with the provided Compose (non-exhaustive):

| Variable | Typical Docker value | Purpose |
|----------|----------------------|---------|
| `PORT` | `80` | HTTP server listen port inside the container |
| `NODE_ENV` | `production` | Conventional Node/Bun environment |
| `DOMAIN` | `instance1`, … | Server identity / URLs for federation and settings |
| `PROTOCOL` | `http` | URL scheme where applicable |
| `DB_PATH` | `/app/data/node1.db`, … | SQLite file path inside the container |
| `JWT_SECRET` | (Compose default in dev) | Signing JWTs (change for real deployments) |
| `ADMIN_USER` / `ADMIN_PASS` | `admin1` / `admin123`, … | Bootstrap admin credentials |
| `ADMIN_DISPLAY_NAME` / `ADMIN_SUMMARY` | optional | Admin profile (see `composition-root.ts`) |

The **`Dockerfile` `EXPOSE 3000`** does not match the federation Compose **`PORT=80`**. That is intentional: the **Compose `command` and `environment` override** the process behavior; `EXPOSE` is documentation for the default image. In the federation stack, each API container listens on **port 80** internally.

## `docker-compose.yml`: nginx + N API instances

There are two ways this file appears:

1. **Checked-in template** — e.g. a fixed number of `instance1` … `instance15` services (as in the repo) with `nginx` depending on all of them.
2. **Generated by `launch.sh`** — same structure, but **N** is configurable.

### Services

- **`nginx`**
  - Image: **`nginx:alpine`**
  - Publishes **`80:80`** on the host.
  - Mounts **`./nginx.conf`** → **`/etc/nginx/nginx.conf`**
  - **`depends_on`** all `instance*` services so startup order is sane.

- **`instance1`**
  - **`build`**: context **`..`** (the `api/` folder), **`dockerfile: docker/Dockerfile`**
  - **`image: microblog-built:latest`** — tags the built image so other instances reuse it.
  - **Environment**, **volumes**, and **`command`** as described above.

- **`instance2` … `instanceN`**
  - **`image: microblog-built:latest`** — no second build.
  - **`pull_policy: never`** — forces use of the locally built tag (Compose v2 behavior).

- **Per-instance volume**: **`./data/instance$i:/app/data`** — SQLite (and any other files under `/app/data`) persist on the host under `api/docker/data/`.

- **Command**: **`bun run dist/server.js`** — matches production entry; redundant with `CMD` but makes Compose explicit.

### Package.json scripts (`api/package.json`)

These use the **legacy** `docker-compose` binary name and point at **`docker/docker-compose.yml`**:

- `docker:up` — `docker-compose -f docker/docker-compose.yml up`
- `docker:down` — `down`
- `docker:build` — `build`
- `docker:logs` — `logs -f`

`launch.sh` uses the **`docker compose`** (V2 plugin) form. Either works if your CLI provides them.

## Federation launcher (`api/docker/launch.sh`)

For local **multi-node federation** testing, **`launch.sh`** automates what would be tedious to hand-edit.

### Usage

```bash
./launch.sh <N>              # Generate docker-compose.yml, nginx.conf, /etc/hosts only
./launch.sh <N> --start      # Same + build image + start all containers
./launch.sh --clear          # Stop/remove containers, volumes, generated files, hosts entries
```

### What it generates

1. **`docker-compose.yml`** (in `api/docker/`)
   - **`nginx`** + **`instance1`…`instanceN`**
   - Only **`instance1`** has a **`build:`** section; others reuse **`microblog-built:latest`** with **`pull_policy: never`**.

2. **`nginx.conf`**
   - One **`upstream instance$i { server instance$i:80; }`** per node.
   - A **default server** on port 80 that proxies **`/`** to **`instance1`** (for requests without a matching `Host`).
   - Per-node **`server { listen 80; server_name instance$i; ... }`** blocks that proxy to the matching upstream, with standard `Host`, `X-Real-IP`, `X-Forwarded-*`, and timeouts.

3. **`/etc/hosts`**
   - Adds **`127.0.0.1 instance$i`** lines tagged with **`# managed:microblog-fednet`** so hostnames resolve locally.
   - **`--clear`** removes those lines with **`sudo sed`**.

### `--start` sequence

1. **`docker compose down --remove-orphans`** in `api/docker/` (best effort).
2. **`docker compose build instance1`** — builds the image once.
3. **`docker compose up -d`** — starts nginx and all API containers.

---

## Networking summary

- **Host → nginx**: port **80** (mapped from the nginx container).
- **nginx → API**: HTTP to **`instance$i:80`** on the Docker network (service DNS names).
- **Browser**: **`http://instance$i`** works after **`hosts`** entries (or equivalent DNS in a real deployment).

This is a **development / federation lab** topology, not a production hardening guide (TLS, secrets management, and resource limits are left to the operator).

---

## Tear-down (`--clear`)

`./launch.sh --clear`:

1. **`docker compose down --remove-orphans --volumes`** (removes named/anonymous volumes tied to the stack).
2. Deletes **`docker-compose.yml`** and **`nginx.conf`** in `api/docker/`.
3. Removes **`# managed:microblog-fednet`** lines from **`/etc/hosts`**.

## Quick reference: build and run without the launcher

From **`api/`**:

```bash
docker build -f docker/Dockerfile -t microblog-api:local .
```

Run (example SQLite path and port):

```bash
docker run --rm -e PORT=3000 -e DB_PATH=/app/data/node.db -v "$(pwd)/data:/app/data" -p 3000:3000 microblog-api:local
```

Adjust **`-e`** and **`-p`** to match; the federation Compose uses **PORT=80** inside the container and **nginx** on host **80**.

## Security notes (defaults)

The generated Compose uses **placeholder** values such as **`JWT_SECRET=change-me-in-production`** and **`ADMIN_PASS=admin123`**. Treat these as **local-dev only**; replace them for any shared or public deployment.
