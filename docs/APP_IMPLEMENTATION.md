# Application implementation overview

This document describes **how the fednet-microblog system is implemented** in realistic terms: architecture, major code paths, data flow, and operational constraints. It complements the root [`README.md`](../README.md) and [`API_DOCKER_DEPLOYMENT.md`](./API_DOCKER_DEPLOYMENT.md).

---

## 1. What you are shipping

The product is a **single deployable unit** in production:

- A **Bun** process runs the **Hono** API and serves the **built React SPA** from `api/public/` (static assets + `index.html` fallback).
- **SQLite** (default) holds all application state in one file (`DB_PATH`). **Postgres** is supported via `connection.ts` when `DATABASE_URL` / `DB_DIALECT` are configured.
- **ActivityPub** federation (WebFinger, actors, inbox/outbox, HTTP Signatures) is implemented in the `apcore` module and wired through the same HTTP server.

The **`client/`** package is not a separate runtime: it is a **Vite + React** app whose production build output is written to **`../api/public`**, so the API process can serve UI and JSON from one origin.

---

## 2. Repository layout (implementation view)

| Path | Responsibility |
|------|----------------|
| `api/src/` | Backend: modules, HTTP adapters, use cases, schema |
| `api/public/` | Production frontend assets (generated by `client` build) |
| `client/src/` | SPA: pages, components, hooks, service layer calling the API |
| `docs/` | Design and operational notes |
| `seeder/` | Optional tooling outside the main server (e.g. load / federation scenarios) |

The API is organized into **bounded contexts** (folders such as `auth/`, `posts/`, `users/`, `socials/`, `apcore/`, `admin/`, `notifications/`, `shared/`). Each typically contains:

- **`ports/in`** — interfaces use cases expose to HTTP (inputs/DTOs)
- **`ports/out`** — interfaces to persistence, external systems, or infrastructure
- **`usecases`** — application logic orchestrating ports
- **`adapters/http`** — Hono routes, controllers, URL maps
- **`adapters/db`** — Kysely repositories and table shapes

**Dependency injection** is not a framework: a single **`composition-root.ts`** constructs repositories, use cases, event handlers, and route factories, then exports `create*Routes()` functions used by `app.ts`.

## 3. Backend entrypoint and HTTP surface

### 3.1 Boot sequence (`server.ts`)

1. Sets **`UV_THREADPOOL_SIZE`** from CPU count.
2. Loads **`app`** from `app.ts` and runs **`ensureSchema()`** then **`initializeAdmin()`** from `composition-root` (DB tables + bootstrap admin if missing).
3. Reads **`PORT`** from the environment and starts **`Bun.serve`** with **`app.fetch`**.

If initialization fails, the process exits with a non-zero status.

### 3.2 Global middleware (`app.ts`)

- **Logger** on all routes.
- **CORS** with `origin: "*"` and `exposeHeaders: ["Link"]` (ActivityPub often uses `Link` headers).
- **`globalExceptionHandler`** for consistent error responses.
- **Static files** under `/assets/*` from `./public` (Vite build output).
- **Route mounting** (order matters for matching):
  - `/api` — health, config, docs, OpenAPI, authenticated **proxy** for remote ActivityPub fetches
  - `/api/auth` — register, login, logout
  - `/api/admin` — admin-only JSON API
  - `/api/users`, `/api/notifications` — authenticated user features
  - `/` — **ActivityPub** routes (WebFinger, actor, inbox, outbox) and **posts/socials** routes mounted at path prefixes that overlap ActivityPub URL patterns (see below)
- **SPA fallback**: `GET *` serves `./public/index.html` so client-side routing works when the user refreshes on a deep link.

### 3.3 URL design: REST vs ActivityPub

- **JSON REST** for the SPA lives under `/api/...`.
- **ActivityPub** uses **well-known paths** on the site root (e.g. `/.well-known/webfinger`, `/u/:username`, outbox/inbox). The **`posts`** module exposes **public note URLs** like `/u/:username/statuses/:id` for **GET** (and authenticated **PATCH/DELETE** on `/:id` in `PostsRoutes`) — aligned with common fediverse URL patterns.

The **same Hono `app`** serves both; controllers serialize **ActivityStreams JSON** where appropriate and **application/json** for the SPA API.

## 4. Configuration and identity

The server derives a canonical **origin** for links and signatures:

- **`PROTOCOL`**, **`DOMAIN`**, **`PORT`** → `host` (includes port when not 80/443) and `ourOrigin` (`protocol://host`).

This origin is used for:

- Actor IDs, object IDs, and inbox/outbox URLs in ActivityPub payloads
- **HTTP Signatures** on outbound requests (`HttpSignatureService` + `FederationDelivery`)
- Resolving local vs remote resources in handlers

**JWT** authentication uses **`JWT_SECRET`** and a **token blacklist** table for logout invalidation. **Bootstrap admin** credentials come from **`ADMIN_*`** env vars on first run.

**Server settings** (registration mode, federation flags, etc.) are stored in **`server_settings`**; some fields are only seeded from env when the row is first created (see README).

## 5. Data layer

### 5.1 Schema (`schema.ts` / `ensureSchema`)

On startup, **`ensureSchema`** runs **`CREATE TABLE IF NOT EXISTS`** (and related DDL) for all modules. There is **no versioned migration system**: operators are expected to test schema changes against copies of production data.

**Tables** (conceptually): users, credentials (password hashes + **RSA key material** for ActivityPub signing), token blacklist, invite tokens, server settings, posts, announces (boosts), likes, follows, notifications.

**Note:** The root README documents that the **`follows`** table may be **recreated** on startup in some configurations to enforce an actor-URL-based schema — this is a **sharp edge** for upgrades and should be treated as a known operational risk.

### 5.2 Database abstraction (`connection.ts`)

**Kysely** is the query layer. **SQLite** uses **Bun’s native SQLite** via `@meck93/kysely-bun-sqlite`. **Postgres** is available via `pg` and dialect-specific compiler patches for auto-increment columns. The rest of the app talks to **repository interfaces** keyed off `DbTables` in `schema.ts`.

## 6. Events and federation pipeline

### 6.1 In-process event bus (`shared/adapters/events/EventBus.ts`)

A lightweight **Node `EventEmitter`**-based bus implements **`IEventBus`**:

- **`emit`** schedules async delivery with **`setImmediate`** and catches handler errors (logs; does not crash the process by default).
- **`emitAndAwait`** awaits all registered async handlers (used where ordering matters).

Subscribers are registered in **`composition-root.ts`**.

### 6.2 Registered events (high level)

| Event | Typical purpose |
|-------|------------------|
| **`post.created`** | After a local post is persisted — **fan-out** to remote inboxes, local notifications |
| **`activity.c2s.submitted`** | Client-to-server ActivityPub POST to outbox — **Follow**, **Create**, **Announce**, **Like** handling |
| **`activity.received`** | Server-to-server POST to inbox — verify signature, apply remote **Follow** / **Create** / **Announce** / **Like** |

Handlers are classes under `apcore/handlers/` (e.g. `HandleC2SCreateActivity`, `HandleS2SCreateActivity`, `HandlePostCreated`).

### 6.3 Outbound federation (`FederationDelivery`)

- Resolves **target inbox** from actor URL (fetch actor, read `inbox`).
- Signs the HTTP request with the **local user’s private key** from the **credentials** store (paired with public key published on the actor).
- Performs **signed `POST`** to remote inboxes.

Failures are logged and may surface as errors depending on caller; the design favors **eventual consistency** (typical for fediverse software).

### 6.4 Inbound federation

- **`POST /u/:username/inbox`** expects an **HTTP Signature**; the stack resolves the sender’s key and validates the signature before dispatching **`activity.received`**.

## 7. Authentication module (summary)

- **Registration / login** hash passwords (bcrypt via configured hasher), issue **JWTs**, store **per-user RSA key pairs** for ActivityPub.
- **`authGuard`** middleware decodes JWT, loads user context, and attaches it to Hono context for protected routes.
- **Invite** registration uses **`invite_tokens`** when `server_settings` requires it.

---

## 8. Posts, social graph, notifications

- **Posts** — CRUD and ActivityPub note serialization; **likes** and **announces** are separate tables linked to posts.
- **Socials** — **follows** graph drives both timeline relevance and **fan-out** lists for delivery.
- **Notifications** — listeners on the event bus write **`notifications`** rows for follows, mentions, likes, etc. (exact behavior is in `notifications/` adapters and handlers).

## 9. Frontend implementation (`client/`)

### 9.1 Stack

- **React 19** + **TypeScript**
- **Vite** — dev server; **production build** outputs to **`api/public`** (`vite.config.ts` `outDir: '../api/public'`, `emptyOutDir: false` to avoid wiping unrelated files)
- **React Router** — `routes/index.tsx` composes auth, user, admin routes; lazy-loaded pages with suspense fallbacks
- **TanStack Query** — server state, caching, mutations in `hooks/queries` and `hooks/mutations`
- **Zustand** — **`authStore`** holds JWT + user profile; coordinates with Query on login/logout
- **MUI** — layout and components

### 9.2 API access

- **`apiClient.ts`** wraps `fetch` to **`API_BASE` + path**, JSON body/headers, and maps error responses to **`Error`** with message extraction.
- **`config.ts`** defines **`API_BASE`**. Using **`window.location.origin`** matches **same-origin** deployment (API serves the SPA). For **local dev** with Vite on another port, you typically either:
  - set **`API_BASE`** to the API origin (e.g. `http://localhost:3000`), or  
  - add a **Vite dev proxy** so the browser stays same-origin, or  
  - run the API and open the UI through the API’s static hosting.

The repo’s root README historically described proxying to `API_BASE`; align **`config.ts`** with how you actually run dev.

### 9.3 Auth in the UI

- Token stored in **`localStorage`**; **`AuthInitializer`** / **`useAuth`** hydrate session on load.
- **`ProtectedRoute`** gates authenticated routes; admin routes also require **`is_admin`**.
- Services under **`services/`** (`auth.ts`, `posts.ts`, `federation.ts`, `admin.ts`, …) encapsulate endpoints; hooks use them with React Query.

### 9.4 Federation and feeds in the client

- **WebFinger** and **proxy** services support resolving remote actors and fetching remote objects without browser CORS issues (server-side proxy with auth).
- **Home feed** logic may combine local/outbox data with ranking helpers (`feedService`, `feedRanking`, etc.) — treat these as **product-specific**; they evolve independently of the core API.

## 10. Admin API

Admin routes are mounted under **`/api/admin`**. They use the same JWT mechanism but enforce **`is_admin`** on the user record. Capabilities include user/post moderation, **server settings**, and **invite tokens** (see `admin` module and client `pages/admin/`).

## 11. Testing and quality

- **`bun test`** runs API tests under `api/tests/`.
- Shell-based integration tests may assume a **running server** (see README).
- **Lint/format** scripts exist for API and client; CI expectations depend on your pipeline.

## 12. Realistic limitations (for implementers)

1. **No migration framework** — schema changes are manual and risky on live DBs.
2. **Follows table caveat** — documented behavior may drop legacy follow data on certain startups; plan backups and tests.
3. **CORS is permissive** (`*`) — acceptable for many self-hosted setups; tighten if you expose the API separately.
4. **Federation is asynchronous and best-effort** — remote servers may be down, signatures may fail, retries may be limited; UX should not assume instant cross-instance consistency.
5. **Secrets** — default Docker Compose values are for **local labs** only.
6. **Single-process** — horizontal scaling would require shared nothing state, shared DB, and careful handling of signing keys and delivery; not described as a first-class feature here.
